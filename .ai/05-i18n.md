# Internationalization Guidelines

## Internationalization Principle

VeryKit is an international product created in Brazil.

The project must not be designed as a Brazilian-only toolkit with translations added later. Internationalization is mandatory from day one.

## Required Library

Use `next-intl` for internationalization.

The implementation should support:

- Server Components.
- Client Components.
- Localized routing.
- Localized metadata.
- Locale detection.
- Locale switching.

## Supported Locales

Initial locales:

- `en` as the default language.
- `pt-BR`.
- `es`.

English is the canonical default language for product structure and international visibility.

## Translation Files

Translations should be centralized under:

```text
messages/
  en.json
  pt-BR.json
  es.json
```

Additional locale files may be added later.

## No Hardcoded Strings

No user-facing string should be hardcoded in components.

This includes:

- Navigation labels.
- Button text.
- Input labels.
- Placeholder text.
- Helper text.
- Empty states.
- Error messages.
- Success messages.
- Tool descriptions.
- Examples.
- FAQ.
- SEO metadata.

Exceptions:

- Internal ids.
- Stable slugs.
- Tool names intentionally kept in English.
- Technical format names such as JSON, XML, JWT, UUID, Base64.

## Tool Names

Tool names do not translate.

Examples:

- `JSON Compare` remains `JSON Compare`.
- `JWT Decoder` remains `JWT Decoder`.
- `CPF Generator/Validator` remains `CPF Generator/Validator`.

Descriptions, examples, FAQ, metadata, and helper text must be translated.

## Locale Routing

Recommended route model:

```text
/en
/en/tools/json-compare
/pt-BR
/pt-BR/tools/json-compare
/es
/es/tools/json-compare
```

The exact implementation may be adjusted during scaffolding, but it must support international SEO and clear locale-specific URLs.

## Locale Detection

Locale detection should consider:

- Saved user preference.
- Browser language.
- Default fallback to `en`.

The project may use cookies, localStorage, or both depending on final routing strategy.

Rules:

- The selected language should persist.
- Users must be able to switch language visually.
- The fallback must be predictable.
- Unsupported browser locales should fall back to `en`.

## Language Switcher

VeryKit must include a language switcher.

Requirements:

- Accessible label.
- Keyboard support.
- Clear current language.
- Persist selection.
- Work in light and dark mode.
- Avoid disrupting the user's current page where possible.

## Message Organization

Recommended message namespaces:

```text
common
navigation
home
about
tools
tools.{toolId}.meta
tools.{toolId}.form
tools.{toolId}.result
tools.{toolId}.examples
tools.{toolId}.faq
seo
errors
```

Example conceptual structure:

```text
tools.jsonCompare.meta.description
tools.jsonCompare.form.inputLabel
tools.jsonCompare.result.copySuccess
tools.jsonCompare.faq.items
```

The exact JSON shape should optimize readability and maintainability.

## Error Messages

Error messages must be translated and mapped from stable error keys.

Avoid returning raw English strings from utility functions.

Preferred direction:

- Utility returns `errorKey`.
- UI maps `errorKey` to localized copy.
- Optional structured details are inserted into the translated message.

## SEO Metadata

Metadata must be localized:

- Page title.
- Page description.
- Open Graph title.
- Open Graph description.
- Twitter title.
- Twitter description.
- Schema.org descriptions where applicable.

Tool names remain stable, but page descriptions change by locale.

## International SEO

Each localized page should support:

- Canonical URL.
- `hreflang` alternates.
- Correct locale metadata.
- Language-specific sitemap entries.

## Regional Tools

Regional tools should be understandable globally.

Example:

- Tool name: `CPF Generator/Validator`.
- English description: "Generate and validate Brazilian CPF numbers for testing."
- Portuguese description: "Gere e valide números de CPF brasileiros para testes."
- Spanish description: "Genera y valida números CPF brasileños para pruebas."

Regional context should be explained in descriptions and FAQ.

## Content Tone

English:

- Clear.
- Direct.
- Global.
- Not overly casual.

Portuguese Brazil:

- Natural Brazilian Portuguese.
- Avoid literal translations.
- Keep technical terms recognizable.

Spanish:

- Neutral Spanish.
- Avoid country-specific idioms where possible.

## Adding a New Locale

To add a locale in the future:

1. Add the locale to the supported locale list.
2. Create a new message file.
3. Translate global navigation and home content.
4. Translate all stable tool metadata.
5. Update SEO alternate links.
6. Update language switcher labels.
7. Verify layout with longer text.

## Translation Review Checklist

Before approving i18n work:

- No user-facing string is hardcoded.
- All supported locales have the required keys.
- Fallback behavior is clear.
- Tool names remain stable.
- Metadata is translated.
- Error messages are translated.
- Longer translated text does not break layout.
- Language switcher remains accessible.
