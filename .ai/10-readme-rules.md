# README Rules

## Purpose

The README is the public front door of VeryKit.

It should help readers quickly understand:

- What VeryKit is.
- Who it is for.
- Why it exists.
- How to run it.
- How to contribute.
- What principles guide it.

## Tone

The README should be:

- Clear.
- Friendly.
- Professional.
- Direct.
- Open source oriented.
- Free of marketing exaggeration.

## Required Sections

The README should include:

- Project name.
- Slogan.
- Short description.
- Product vision.
- Feature/category overview.
- Technology stack.
- Local development instructions.
- Project structure overview.
- Internationalization note.
- Accessibility note.
- Performance goals.
- Contributing link.
- Code of conduct link.
- Security policy link.
- License.

## Product Positioning

The README must communicate:

- VeryKit is free.
- VeryKit is open source.
- VeryKit is made for QA Engineers and Developers.
- VeryKit has no ads.
- VeryKit has no forced login.
- VeryKit is privacy-conscious.
- VeryKit is international.

## Avoid

Do not include:

- Commercial sales copy.
- Pricing language.
- Roadmap promises that sound guaranteed.
- Claims that are not true yet.
- Setup instructions that are not verified.
- Badges that imply unavailable CI or releases.

## Installation Instructions

During Phase 1, the README should state that application setup is not available yet because the project is in documentation/architecture phase.

After implementation begins, setup instructions should be accurate and include:

```text
git clone
npm install
npm run dev
npm run test
npm run lint
```

Exact package manager commands should match the repository.

## Feature List Rules

Feature lists should distinguish:

- Planned tools.
- Implemented tools.
- Beta tools.

Do not imply a tool exists before it is implemented.

## Project Structure Rules

The README may show a simplified project structure.

It should avoid duplicating all details from `.ai/01-architecture.md`.

## Contribution Rules

The README should link to `CONTRIBUTING.md`.

It should make contribution feel welcome while preserving the quality bar.

## Governance Links

The README should link to:

- `CONTRIBUTING.md`.
- `CODE_OF_CONDUCT.md`.
- `SECURITY.md`.
- `CHANGELOG.md`.
- `LICENSE`.

## Maintenance Rule

The README must be updated when:

- Setup commands change.
- Tool status changes.
- Supported locales change.
- Major architecture changes occur.
- Contribution process changes.
- Deployment details change.

## README Review Checklist

Before approving README changes:

- It accurately reflects the current phase.
- It does not overclaim implemented features.
- It links to governance files.
- It explains the product clearly.
- It keeps the zero ads/login/paywall principle visible.
- It is friendly to contributors.
- It avoids unnecessary length.
