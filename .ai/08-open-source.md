# Open Source Guidelines

## Open Source Vision

VeryKit is a 100% open source project built to help QA Engineers, developers, automation engineers, students, and technical communities.

The project should be useful, welcoming, technically strong, and easy to contribute to.

## License

VeryKit uses the MIT License.

This allows broad use, modification, distribution, and contribution while keeping the project simple for individuals, companies, educators, and communities.

## Maintainer Values

Maintainers should optimize for:

- Product usefulness.
- Code quality.
- Accessibility.
- Internationalization.
- Performance.
- Contributor clarity.
- Respectful communication.
- Sustainable project scope.

## Community Principles

VeryKit should be:

- Friendly to first-time contributors.
- Serious about quality.
- Clear about expectations.
- Respectful in review.
- Transparent about decisions.
- Safe for users.
- Free from commercial pressure.

## Contribution Types

Valuable contributions include:

- New tools.
- Bug fixes.
- Accessibility improvements.
- Translation improvements.
- Documentation improvements.
- Test coverage.
- Performance improvements.
- SEO improvements.
- Design consistency fixes.
- Refactors that reduce real complexity.

## Contribution Quality Bar

Contributions should:

- Follow architecture guidelines.
- Keep tools isolated.
- Include relevant tests.
- Include translations.
- Preserve accessibility.
- Avoid unnecessary dependencies.
- Avoid unrelated changes.
- Explain the user value.

## Pull Request Expectations

Every pull request should include:

- Clear summary.
- Motivation.
- Screenshots or recordings for UI changes when applicable.
- Testing notes.
- Accessibility notes when relevant.
- i18n notes.
- Dependency notes.

## Review Culture

Reviews should be:

- Direct.
- Kind.
- Specific.
- Focused on the code and product.
- Clear about blocking vs non-blocking feedback.

Reviewers should avoid vague criticism. Contributors should avoid taking technical feedback personally.

## Issue Policy

Issues should be used for:

- Bug reports.
- Feature requests.
- Tool proposals.
- Documentation improvements.
- Accessibility reports.
- Security reports should follow `SECURITY.md`, not public issues.

## Tool Proposal Template Direction

A tool proposal should explain:

- Tool name.
- User problem.
- Target category.
- Universal or regional scope.
- Example inputs and outputs.
- Privacy considerations.
- Similar existing tools.
- SEO keywords.
- Accessibility considerations.

## First-Time Contributors

The project should maintain issues appropriate for new contributors when possible.

Good first issues may include:

- Translation improvements.
- Documentation fixes.
- Small test cases.
- Simple utility edge cases.
- Accessibility label improvements.

## Translation Contributions

Translation PRs should:

- Preserve tool names.
- Avoid literal translations when unnatural.
- Keep technical terms clear.
- Check UI layout with longer text.
- Include all required message keys.

## Dependency Contributions

Adding a dependency requires strong justification.

The PR must explain:

- Why native APIs are insufficient.
- Bundle impact.
- Maintenance status.
- Security posture.
- Alternatives considered.

## Security Contributions

Security vulnerabilities should be reported privately through the process in `SECURITY.md`.

Do not disclose exploitable issues publicly before maintainers have had a reasonable chance to respond.

## Governance Direction

VeryKit should start with lightweight governance.

Recommended roles over time:

- Maintainers.
- Core contributors.
- Translation contributors.
- Security contacts.

Governance can evolve as the community grows.

## Release Direction

Releases should be documented in `CHANGELOG.md`.

Release notes should include:

- Added tools.
- Changed behavior.
- Fixed bugs.
- Accessibility improvements.
- Performance improvements.
- Security notes where appropriate.

## Open Source Checklist

Before public launch:

- README is complete.
- CONTRIBUTING is complete.
- CODE_OF_CONDUCT is present.
- SECURITY policy is present.
- LICENSE is present.
- CHANGELOG is initialized.
- Architecture docs are present.
- Tool contribution guidance is clear.
- Local setup instructions are accurate.
